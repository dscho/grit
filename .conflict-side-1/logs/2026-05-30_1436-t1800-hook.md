## Fix 43 — post-update hook on local push
The local push path ran pre-receive, reference-transaction
(preparing/prepared/committed) and post-receive but never post-update.
Added a post-update invocation immediately after the post-receive block,
guarded by `!args.dry_run && !applied_updates.is_empty()`. It passes the
updated remote refnames as argv (not stdin), matching receive-pack
(run_update_post_hook) and receive_pack.rs:736-756; its exit status is
ignored because post-update is informational. Output is colorized via
colorize_remote_output like the other remote hooks.

## Fix 44 — push-to-checkout hook on receive.denyCurrentBranch=updateInstead
grit always ran the diff-files/diff-index cleanliness check and an internal
`checkout --force`, never consulting the push-to-checkout hook. Real git's
update_worktree() (git/builtin/receive-pack.c:1475-1498) gives the hook first
refusal (run with GIT_WORK_TREE + GIT_DIR set and the new commit hash as the
sole argv) and only falls back to push_to_deploy (cleanliness check + read-tree)
when the hook was NOT invoked.

Changes in push.rs:
- check_receive_pack_policy: the UpdateInstead arm no longer eagerly runs
  worktree_clean_for_update_instead. Worktree handling moved entirely into the
  post-ref-write worktree-update step.
- update_worktree_after_push_update_instead: now runs the push-to-checkout hook
  first (via run_hook_in_git_dir with GIT_DIR/GIT_WORK_TREE env). If the hook
  ran (was_executed): a non-zero exit rejects the ref ("push-to-checkout hook
  declined"); a zero exit means the hook already updated the worktree, so the
  cleanliness check + internal checkout are skipped. Only when the hook was
  NotFound do we fall back to worktree_clean_for_update_instead +
  update_worktree_after_push_update_instead_checkout (the prior checkout logic,
  extracted into its own fn). Returns captured hook output so the caller can
  colorize/emit it.
