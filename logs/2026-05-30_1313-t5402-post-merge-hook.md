# t5402-post-merge-hook — fix log (2026-05-30)

## Result
- Before: 3/7 passing. After: 7/7 passing.
- Failing subtests fixed: 5402.4, 5402.5, 5402.6, 5402.7.

## Root cause
All four failures shared one root cause in `grit-lib/src/hooks.rs::hook_argv0`.

The tests run `GIT_DIR=clone1/.git git merge ...` (and `clone2/.git` for squash) from
the trash dir, without setting `GIT_WORK_TREE`. Per `repo.rs:287`, grit then defaults
`repo.work_tree` to the process cwd (`<trash>`), while `repo.git_dir` becomes the
absolute `<trash>/clone1/.git`. So `git_dir` is NOT `work_tree/.git`.

`hook_argv0` relativised the traditional-hook argv0 to `.git/hooks/<name>` whenever
`cwd == repo.work_tree`. With work_tree == `<trash>`, that argv0 resolved against
`<trash>/.git/hooks/post-merge` — which does not exist (the real hook is under
`<trash>/clone1/.git/hooks/post-merge`). exec() returned ENOENT, the hook never ran,
and `clone*/.git/post-merge.args` was never written. Subtests 5 and 7 are pure
`grep` dependencies on that args file.

(`report_spawn_error` printed the absolute `path`, which masked that the actual
spawned argv0 was the broken relative path.)

## Fix
In `grit-lib/src/hooks.rs::hook_argv0`, gate the `.git/hooks/<name>` relativisation
branch so it only fires when `repo.git_dir` is literally `work_tree/.git`
(symlink-safe comparison via a new `same_dir` helper that falls back to
`canonicalize`). When that condition is false, fall through to the absolute
`hooks_dir.join(hook_name)`. This mirrors upstream `git/hook.c`
(`list_hooks_add_default`), which absolutises the hook path whenever the hook is run
in a specific directory.

The bare-repo branch (`cwd == repo.git_dir` -> `hooks/<name>`) is unchanged; there the
cwd really is the git dir so the relative path is correct.
