## Change made (owned module: grit/src/commands/cherry_pick.rs)
`error_if_cherry_pick_would_clobber_worktree` previously aborted the pick when ANY
path in the parent..HEAD diff was dirty in the worktree — including a path that the
picked commit never rewrites (e.g. a file the user locally renamed). The three-way
merge keeps "ours" at such a path and leaves the worktree file untouched, so it must
not block. Fix: intersect the touched-path set with the paths the picked commit
changes relative to its parent (`diff_trees(parent, theirs)`), so only paths the pick
actually writes can trigger the overwrite guard. This is exactly git's semantics
(merge-recursive only checks verify_uptodate for paths it will update).

Genuine conflicts (you locally edit a file the pick also edits) still block, because
that path IS in the theirs-vs-parent diff. Verified no regressions across the whole
cherry-pick / revert / conflict family.
