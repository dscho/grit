## Root causes found
1. `pack-objects --revs` (the `collect_pack_objects_from_rev_stdin_lines` path) did NOT
   honor `--local`, `--honor-pack-keep`, or `--incremental`. These flags must omit objects
   that are available in non-local (alternate) storage, in `.keep`-marked packs, or in any
   pack respectively — mirroring `want_found_object` / `want_object_in_pack_mtime` in
   `git/builtin/pack-objects.c`.
   - Fix: added `pack_objects_locality_excludes` + `apply_locality_excludes` in
     grit/src/commands/pack_objects.rs, applied to both `--revs` return paths (sparse + default).
   - `--local`: exclude objects loose in an alternate source OR present in a non-local pack.
   - `--honor-pack-keep`: exclude objects in a pack with a `.keep` sidecar.
   - `--incremental`: exclude objects present in any pack (local or alternate).
   - Result: 151 -> 172.

2. Pack bitmaps themselves are NOT really implemented: pack-objects writes an EMPTY
   placeholder `.bitmap` file (std::fs::write(..., [])). So:
   - test 5/81/162 "full repack creates bitmaps" need trace2 keys
     num_selected_commits=106 / num_maximal_commits=107 (bitmap commit-selection algorithm).
   - `rev-list --test-bitmap` is a no-op returning success — so "truncated/corrupt bitmap
     fails gracefully" + "load corrupt bitmap" + "complains about multiple pack bitmaps"
     can't emit the expected error text.
   - `test-tool bitmap list-commits` does not exist -> pack.preferBitmapTips tests fail.
   - "enumerating progress counts pack-reused objects" needs real pack-reuse + bitmaps.
   These require a real pack-bitmap reader/writer (large feature).
